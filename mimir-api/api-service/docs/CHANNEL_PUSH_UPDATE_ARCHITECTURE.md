# Channel Push Update Architecture Proposal

## Objective
Enable channel plugins (e.g., Spotify) that can receive real-time external updates to push those updates into the system, allowing scenes to opt into either:
- Scheduled (pull) updates (current behavior), or
- Real-time push-driven updates provided by the channel itself.

This provides lower latency, fewer API calls, and better UX for time-sensitive content like "Now Playing" music data, while preserving backwards compatibility.

---
## Terms
| Term | Meaning |
|------|---------|
| Channel | A plugin that produces content for displays/scenes |
| Scene | A configured set of content sources rendered to displays |
| Update Strategy | Mode controlling how scene gets fresh data (scheduler vs push) |
| Dispatcher | Internal service brokering push events to subscribers |
| Event | Structured payload representing a channel state change |

---
## High-Level Flow
```
External Source (e.g., Spotify Web API / Webhook / Long Poll)
        ↓
 Channel Plugin (push-capable)
        ↓ emits ChannelUpdateEvent
 ChannelEventDispatcher (dedupe, throttle, cache last)
        ↓
 Distribution Pipeline (same endpoint as scheduler path)
        ↓
 Displays (WebSocket / MQTT / Poll consumers)
```

Scheduler Path (unchanged):
```
APScheduler Job → Channel Fetch → Distribution Pipeline → Displays
```

Both converge at a unified distribution entrypoint.

---
## Update Modes Enumeration
Proposed enum (Python + API):
- `scheduler` (pull only)
- `push` (push only)
- `dual` (supports both; user chooses)
- `push_preferred` (dual but UI highlights push)

Internally we can normalize to capability flags + preferred.

## Channel Capability Contract Additions
Channel metadata (example fragment from /api/v1/channels):
```json
{
  "id": "spotify_status",
  "name": "Spotify Status",
  "capabilities": {
    "update_modes": ["scheduler", "push"],
    "preferred_mode": "push",
    "push_event_types": ["now_playing_changed"],
    "supports_fallback_poll": true
  }
}
```

Channel base interface augmentation (conceptual):
```python
class BaseChannelPlugin(ABC):
    supports_push: bool = False
    push_preferred: bool = False

    # Option 1: async generator streaming updates
    async def stream_updates(self) -> AsyncIterator[ChannelUpdateEvent]:
        raise NotImplementedError

    # Option 2: imperative dispatch registration
    def register_dispatcher(self, dispatcher: ChannelEventDispatcher) -> None:
        pass
```

Only one push style must be implemented.

---
## ChannelUpdateEvent Model (Draft)
```python
class ChannelUpdateEvent(BaseModel):
    channel_id: str
    event_type: str  # e.g. "now_playing_changed"
    payload: dict    # normalized content data
    ts: float        # epoch seconds
    schema_version: int = 1
    hash: str | None = None  # optional dedupe token
```

Hash is generated by stable JSON serialization of `payload` + event_type.

---
## Event Dispatcher Responsibilities
1. Maintain subscribers per (channel_id → {scene_id → callback})
2. De-duplicate sequential identical events
3. Apply throttling / coalescing (token bucket)
4. Cache last event per channel for late subscribers
5. Forward accepted events to:
   - Cache layer (channel data store)
   - Distribution pipeline (existing rendering path)
   - Optional external broadcast (MQTT/Redis) phase 2
6. Emit metrics + structured logs

Interface (conceptual):
```python
class ChannelEventDispatcher:
    def subscribe(self, channel_id: str, scene_id: str, callback: Callback): ...
    def unsubscribe(self, channel_id: str, scene_id: str): ...
    async def publish(self, event: ChannelUpdateEvent): ...
```

## Scene Model Changes
Add fields:
```json
{
  "update_strategy": "scheduler" | "push",
  "poll_interval_seconds": 60,
  "push_fallback_poll_seconds": 120
}
```
Migration:
1. Add nullable `update_strategy` (default `scheduler`)
2. Add nullable `push_fallback_poll_seconds`
3. Backfill existing rows

Behavior:
- If strategy = `push` AND channel supports push → do not register recurring APScheduler job
- If channel does NOT support push but scene configured push → automatically downgrade to scheduler (log + UI warning)

---
## Subscription Lifecycle
On scene activation/start:
1. Resolve channel and strategy
2. If push: register callback with dispatcher
3. Immediately deliver last cached event (if present)
4. Start fallback timer (if configured)
5. If no events within fallback window → trigger one-off fetch, reset timer

On scene deactivation/removal:
- Unsubscribe from dispatcher

---
## Distribution Unification
Introduce a single function:
```python
async def apply_channel_update(channel_id: str, event: ChannelUpdateEvent | None = None):
    """Normalize event or fetched data, update caches, fan-out to displays."""
```
Scheduler flow calls it with data from fetch.
Dispatcher flow calls it with push event.

---
## Fallback & Reliability
Failure scenarios:
| Scenario | Mitigation |
|----------|------------|
| Plugin silence / upstream idle | Fallback poll after N seconds |
| Repeated identical events | Dedupe by hash |
| Burst flood | Token bucket throttle + last-event replacement |
| Plugin crash | Supervisor task restart (limited retries) |
| Schema mismatch | Pydantic validation + reject + log |

Metrics (OpenTelemetry):
- `channel_push_events_total{channel_id}`
- `channel_push_events_dropped{reason}`
- `channel_push_latency_seconds` (publish→distribution)
- `scene_push_fallbacks_total{channel_id}`

---
## Logging Conventions
Structured log keys:
- channel_id, scene_id, event_type, event_hash, duration_ms, dropped_reason
Avoid payload dumps at info; include size or hash only.

---
## UI Exposure (Future Web Layer)
When editing/creating a scene:
1. Fetch channel metadata
2. If `supports_push` → show toggle (Radio: Scheduler vs Real-time Push)
3. If Push selected → optional fallback interval input
4. Display small helper text: "Push delivers instant updates (recommended for Spotify)."

---
## Rollout Phases
Phase 1 (Backend core):
- Event model
- Dispatcher (in-memory)
- Channel base interface update
- Modify a single plugin (Spotify) to emit push updates
- Scene model + migration
- Distribution unification

Phase 2 (API & UI):
- Expose capabilities in channel listing
- Scene create/update accepts `update_strategy`
- Return active strategy in scene detail

Phase 3 (Resilience & External Broadcast):
- Add metrics & throttling
- Optional MQTT topic publish: `mimir/channel/<channel_id>/updates`
- Redis pub/sub or stream for multi-process scale

Phase 4 (Enhancements):
- Multi-event-type selective subscription
- Dead letter queue for rejected events
- Backpressure adaptive polling fallback

---
## Trade-offs
| Decision | Benefit | Cost |
|----------|---------|------|
| In-memory dispatcher first | Low complexity, fast iteration | No horizontal scale |
| Async generator push API | Natural streaming semantics | Some plugins easier with callbacks |
| Unified distribution path | Consistency, less duplication | Minor refactor upfront |
| Fallback polling | Reliability guarantees | Extra timer logic |

---
## Risks & Mitigations
| Risk | Mitigation |
|------|------------|
| Misconfigured scene picks push unsupported channel | Validate at scene save | 
| Event storms (scrubbing) | Throttle + coalesce |
| Plugin blocking in stream | Run each stream in supervised task |
| Schema drift | Versioned event schema |

## Summary
This design introduces an optional push-driven update path for channels while preserving existing scheduled polling. It centralizes event handling through a dispatcher, unifies distribution, and adds minimal metadata to expose capabilities to the UI. Incremental rollout minimizes risk and allows early value via the Spotify channel.

## Implementation Snapshot (Status – 2025-09)

Built (implemented in codebase):
1. Event Model & Dispatcher
  - `app/services/channel_events.py` defines `ChannelUpdateEvent` and a singleton in-memory dispatcher.
  - Basic per-channel last-event awareness (lightweight); full token-bucket throttling not added yet.
2. Push-Capable Plugin (Spotify)
  - Spotify channel in separate repo emits push events and declares `supports_push`.
  - Plugin discovery (`plugin_discovery_service`) registers dispatcher callbacks automatically.
3. Channel Event Consumer
  - `app/services/channel_event_consumer.py` maps channel events to active scenes with `update_strategy=push`.
  - Debounce per scene using `settings.push_debounce_seconds`.
  - Maintains channel→scene mapping cache with TTL (`settings.push_channel_scene_cache_ttl`).
  - Fallback stale poll loop triggers refresh when a push scene is silent beyond its `push_fallback_poll_seconds`.
4. Scene Refresh Service
  - `SceneRefreshService` unifies scheduler and push-triggered refresh paths.
  - Iterates all configured channels in a scene (multi-channel support) instead of first-only.
  - Computes content hash (SHA256) + increments `content_epoch` only on change to reduce unnecessary downstream work.
5. Scene Model & API Fields
  - Added `update_strategy` and `push_fallback_poll_seconds` (migration exists; must be applied in deployed DB).
6. Configuration Tunables (in `config.py`)
  - `PUSH_DEBOUNCE_SECONDS`
  - `PUSH_CHANNEL_SCENE_CACHE_TTL` / `CHANNEL_SCENE_CACHE_TTL_SECONDS`
  - `PUSH_FALLBACK_STALE_CHECK_INTERVAL` / `PUSH_STALE_CHECK_INTERVAL_SECONDS`
7. Reliability Enhancements
  - Fallback stale poller (background task) to guarantee eventual refresh.
  - Lazy import inside `scene_refresh_service` to break circular dependency with plugin discovery.
8. Minimal Metrics Placeholders
  - Counters scaffolded (actual rich metrics/export pending).

Partially implemented or deviated from original proposal:
1. Throttling & Dedupe
  - Proposal: token bucket + dispatcher-level dedupe by hash.
  - Current: scene-level debounce; hashing used post-refresh to detect changed output; dispatcher does not yet perform full hash-based event drop.
2. Unified Distribution Entry Function
  - Proposal: single `apply_channel_update` entrypoint.
  - Current: Logic embodied in `SceneRefreshService.refresh_scene`; not a separately named public function yet.
3. Metrics Depth
  - Proposal: detailed counters + histograms (latency, drops, fallback counts).
  - Current: Minimal placeholders only (no labeled telemetry surfaced).
4. External Broadcast (MQTT/Redis for channel events)
  - Not implemented; only internal in-memory dispatch used.
5. Per-Event-Type Subscription
  - Deferred (all push scenes receive any event for their channel).

Known gaps / next steps (updated):
1. Apply Alembic migration in all environments (resolve "no such column: scenes.update_strategy").
2. Add rich metrics: event counts (accepted/dropped), refresh durations, fallback invocations.
3. Dispatcher-level dedupe + optional token-bucket throttling.
4. Optimize distribution: skip publish when content hash unchanged (already have hash to support this).
5. Graceful shutdown coordination: cancel stale poller tasks explicitly.
6. UI exposure for selecting `update_strategy` and configuring fallback interval.
7. Optional external event fan-out (e.g. MQTT topic `mimir/channel/<channel_id>/updates`).
8. Structured error taxonomy and tighter exception granularity in refresh path (currently pragmatic broad catch at boundary).
9. Per-event-type selective subscription (only if needed for high-volume plugins).
10. Retry / backoff supervision for plugin push streams if future channels use long-lived connections.

Deviation summary (why acceptable now):
- Scaling requirements are modest; in-memory dispatcher + debounce is enough for single-node deployment.
- Implementing hashing earlier enabled future optimization without premature complexity.
- Postponing external broadcast and full metrics reduced initial risk and sped up validation.

Operational note:
- If you encounter the missing column error, run the latest migrations (e.g. `alembic upgrade head`) before enabling push scenes in production.
